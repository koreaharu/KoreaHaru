# I18N — 다국어 전략 (현재 연기됨)

## 현재 상태

**다국어 라우팅은 의도적으로 연기됨.** 영어 단일 사이트로 운영하며, URL에는 locale prefix 없음 (`/`, `/services` 등).

- 라우터: 평면 구조 — `app/page.tsx`, `app/services/page.tsx`
- 메시지 로딩: `next-intl` 인프라는 유지하지만 단일 locale (`en`) 고정
- `messages/en.json`만 존재. 다른 언어 파일은 추후 추가

## 왜 연기했나

- SEO/콘텐츠 우선순위가 더 높음 — 영어 콘텐츠 깊이부터 확보
- 다국어 라우팅은 SEO 최적화(hreflang, 중복 콘텐츠 처리)와 함께 한 번에 처리하는 것이 깔끔
- 번역 품질을 보장할 수 없는 단계에서 placeholder 다국어를 노출하면 브랜드 신뢰 훼손

## 인프라 유지 이유

- 모든 사용자용 텍스트는 이미 `messages/en.json`에 키로 분리되어 있음
- `useTranslations()` 훅으로 컴포넌트가 키 참조 — 나중에 언어 추가가 단순 작업으로 줄어듦
- 라우팅만 추가하면 됨 (`middleware.ts` + `[locale]` 세그먼트)

## 향후 다국어 추가 시 작업 순서

1. `messages/{locale}.json` 파일 추가 (en.json 카피 → 번역)
2. `middleware.ts` 부활 (`localePrefix: 'always'`)
3. `app/` → `app/[locale]/` 으로 페이지 이동
4. 컴포넌트에 `locale` prop 다시 추가하거나 `useLocale()` 훅 사용
5. `lib/seo.ts`에 `hreflang` / `languages` alternates 추가
6. `i18n.ts`에 다중 locale 로딩 복원

## 향후 지원 예정 locale

| 코드 | 언어 | 우선순위 |
|---|---|---|
| `en` | English | 현재 (유일) |
| `ja` | Japanese | Phase 2 |
| `zh-TW` | Traditional Chinese | Phase 2 |
| `zh-CN` | Simplified Chinese | Phase 2 |
| `vi` | Vietnamese | Phase 3 |
| `th` | Thai | Phase 3 |
| `ko` | Korean | Phase 3 |

## 키 네이밍 규칙

`<namespace>.<section>.<key>`

예시:
- `home.hero.title`
- `services.chauffeur.title`
- `common.button.book_now`

## 규칙 (지금부터 지켜야)

- 모든 사용자용 텍스트는 메시지 키로 분리 — **하드코딩 금지**
- 새 컴포넌트 작성 시 `useTranslations()` 사용
- 메시지 키는 영어 평문이 아닌 시맨틱 식별자 (`home.hero.title`, NOT `a_curated_journey_through_southern_korea`)
- 날짜/통화/숫자는 `Intl` API 사용
